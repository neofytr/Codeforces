1. 

Let n >= 1
Let a[1, n] be an array of non-negative integers that each fit in 64 bits.

Suppose we want to find the following:
A = max(1 <= i <= j <= n, a[i] ^ ... ^ a[j])

Let p[0] = 0 and p[i] = p[i - 1] ^ a[i] for 1 <= i <= n
Then, A = max(1 <= i <= j <= n, p[j] ^ p[i - 1])

The standard way to find A is using binary trie:

--------------------------------------------------------
// each trie node stores two children:
// nxt[0] -> bit 0
// nxt[1] -> bit 1
struct Node {
	Node *nxt[2];

	Node() {
		nxt[0] = nxt[1] = nullptr;
	}
};

class BinaryTrie {
private:
	Node *root;

public:
	BinaryTrie() {
		root = new Node();
	}

	// inserts a number into the trie
	void insert(int x) {
		Node *cur = root;

		// we go from most significant bit -> least significant bit
		// because larger bits contribute more to the xor value
		for (int b = 62; b >= 0; b--) {

			// extract current bit
			int bit = (x >> b) & 1;

			// create path if it doesn't exist
			if (cur->nxt[bit] == nullptr)
				cur->nxt[bit] = new Node();

			// move downward
			cur = cur->nxt[bit];
		}
	}

	// returns maximum possible value of (x ^ y)
	// where y is some number already inserted
	int max_xor(int x) {
		Node *cur = root;

		int ans = 0;

		// again process bits from msb -> lsb
		for (int b = 62; b >= 0; b--) {

			int bit = (x >> b) & 1;

			// for xor:
			// 0 ^ 1 = 1
			// 1 ^ 0 = 1
			//
			// so ideally we want opposite bit
			int want = bit ^ 1;

			// if opposite bit exists,
			// we can make this xor bit = 1
			if (cur->nxt[want] != nullptr) {

				// set this bit in answer
				ans |= (1LL << b);

				// move to preferred branch
				cur = cur->nxt[want];
			}

			// otherwise we are forced to take same bit
			else {
				cur = cur->nxt[bit];
			}
		}

		return ans;
	}
};

signed main() {
	int n;
	cin >> n;

	vector<int> a(n + 1);

	for (int i = 1; i <= n; i++)
		cin >> a[i];

	BinaryTrie trie;

	// prefix xor
	int px = 0;

	// important:
	// insert p[0] = 0 first
	//
	// because subarrays can start from index 1
	trie.insert(0);

	int ans = 0;

	for (int i = 1; i <= n; i++) {

		// current prefix xor
		px ^= a[i];

		// find best previous prefix xor
		// that maximizes:
		//
		// px ^ previous_prefix
		//
		// which equals some subarray xor
		ans = max(ans, trie.max_xor(px));

		// now insert current prefix xor
		// so future positions can use it
		trie.insert(px);
	}

	cout << ans << '\n';
}

2. If the order of queries doesn't matter (queries are offline), think of reordering the queries in a manner that make the solution easier

3. 

When doing DP/string problems where transitions depend on substrings ending at position i:

	dp[i] depends on substrings s[j...i]

we usually iterate:

	for (int j = i; j >= 1; j--)

Trying to build the substring manually is inefficient because:

	tmp = s[j] + tmp

requires insertion at the front of the string, which is costly.

A better approach is to store all dictionary/pattern strings in a trie in REVERSED order.

Example:
	"abc" is stored as:
		c -> b -> a

Now while iterating backwards from i to 1:

	for (int j = i; j >= 1; j--)

we can walk the trie directly using s[j].

This naturally matches the reverse traversal order and avoids:
	- building temporary substrings
	- front insertions
	- repeated hashing/comparisons

Pattern:
	- substring ends at i
	- iterate j backwards
	- store dictionary strings reversed in trie
	- traverse trie while moving j backwards

This converts substring existence checks from:
	O(length of substring construction/check)

to:
	O(1) trie transition per character.

Very common in:
	- word break DP
	- dictionary DP
	- suffix-based transitions
	- substring existence queries
	- automaton/string DP problems

4. 

Usually in tree DP, we first optimize the child subtree before solving for the parent (post-order traveral). We can also solve for the parent first and then recurse to solve for the children (pre-order traversal). Prove and choose the one which is correct.

5. DSU Successor / Predecessor Compression Trick
=============================================

1. Problem

Maintain a subset S ⊆ U where:
	U = {0,1,2,...,n}
under permanent deletions while supporting either:

Predecessor queries:
	pred(x) = max { y ∈ S | y ≤ x }

or successor queries:
	succ(x) = min { y ∈ S | y ≥ x }

Target complexity:
	O(α(n)) amortized per operation.

2. Predecessor Structure

Invariant:
	find(x) = pred(x)

Representation:
	parent[x] stores the next candidate to check for pred(x).

Initially:
	S = U
	parent[x] = x

Thus:
	find(x) = x

meaning x is alive.

3. Deletion

Suppose x is deleted.
Deletion basically binds a chain from x to x - 1.

Observation:
	pred(x) = pred(x-1)

Therefore:
	parent[x] = find(x-1)

Interpretation:
	x redirects to the largest alive element below it.

4. Query

find(x):
	if parent[x] == x:
		return x

	return parent[x] = find(parent[x])

Path compression preserves:
	find(x) = pred(x)

while flattening chains.

5. Example

Initially:
	S = {1,2,3,4,5,6,7}

Delete 5:
	parent[5] = find(4) = 4

Delete 4:
	parent[4] = find(3) = 3

Then:
	find(5)
	→ find(4)
	→ find(3)
	= 3

Thus:
	pred(5) = 3

After compression:
	parent[5] = 3

6. Sentinel

Use:
	0

as dummy null node.

Then:
	find(x)=0

iff no alive element ≤ x exists.

7. Successor Variant

Invariant:
	find(x)=succ(x)

Deletion:
	parent[x]=find(x+1)

Sentinel:
	n+1

8. Correctness

Invariant:
	find(x) always equals the nearest alive candidate.

Base:
	initially every element is alive,
	thus find(x)=x.

Induction:
	after deleting x,

	best alive ≤ x
	=
	best alive ≤ x-1
	=
	find(x-1)

thus:
	parent[x]=find(x-1)

preserves the invariant.

9. Complexity

With path compression:
	find   : O(α(n)) amortized
	delete : O(α(n)) amortized

No union-by-rank is needed since edges are directional.

10. Applicability Conditions

This technique applies when:
1. Universe is contiguous.
2. Updates are monotonic (permanent deletions only).
3. Queries are predecessor/successor type.
4. Reinsertion is not needed.

11. Common Forms

- largest unused ≤ x
- smallest unused ≥ x
- next free slot
- next unvisited index
- skip processed positions
- next valid state/index

12. Typical Applications

- scheduling
- greedy assignment
- interval painting
- offline range processing
- parking/room allocation
- skipping deleted indices

13. Limitations

Does NOT support:
- reinsertion
- arbitrary insertions
- kth order statistics
- fully dynamic ordered set operations

Use balanced BST / PBDS / Fenwick / segment tree for those.
