81. Counting Intersecting Segments with a Sliding Window in O(1) per shift

Problem setup:
- You are given multiple segments [l, r].
- You slide a fixed-length window [x, x + d − 1] across a line.
- For each window position, you want to count how many segments intersect it.

Key observation:
A segment [l, r] intersects the window [x, x + d − 1] iff:
    l ≤ x + d − 1  AND  r ≥ x

Instead of checking this condition for every segment at every window position
(which would be O(n·k)), we convert the problem into incremental updates.

------------------------------------------------------------
CORE IDEA: EVENT-BASED COUNTING WITH PREFIX EVENTS
------------------------------------------------------------

We track how the intersection count changes when the window shifts by 1.

Define two arrays:
- left[i]  = number of segments that start at position i
- right[i] = number of segments that end at position i

Initial window:
- Window = [1, d]
- Count all segments with l ≤ d
  (these are exactly the segments that intersect the initial window)

Sliding the window:
- Suppose the current window is [x, x + d − 1]
- Next window is [x + 1, x + d]

What changes?

1. Segments that START at (x + d)
   - These now intersect the window
   - Contribution: +left[x + d]

2. Segments that END at x
   - These no longer intersect the window
   - Contribution: −right[x]

So the update rule is:
    cnt += left[x + d]
    cnt -= right[x]

This update is O(1).

------------------------------------------------------------
WHY THIS WORKS
------------------------------------------------------------

Each segment contributes:
- +1 exactly when its left endpoint enters the window
- −1 exactly when its right endpoint leaves the window

Thus, every segment is processed only twice overall:
- Once when it becomes active
- Once when it becomes inactive

------------------------------------------------------------
TIME & SPACE COMPLEXITY
------------------------------------------------------------

Preprocessing left[] and right[]:  O(k)
Initial count:                     O(k)
Sliding window updates:            O(n)
Total time:                        O(n + k)
Extra space:                       O(n)

------------------------------------------------------------
GENERAL APPLICATIONS
------------------------------------------------------------

This technique applies whenever:
- You slide a fixed-size window over an index line
- You need to count how many intervals / events / ranges intersect the window

Common use cases:
- Interval overlap queries
- Scheduling problems
- Maximum or minimum overlap in fixed-length ranges
- Sweep-line problems reduced to 1D sliding windows

Key pattern to remember:
"Convert interval intersection checks into +1 / −1 events at boundaries,
then maintain a running count with O(1) updates."

82.

Problem:
Given an array A of length n (n ≥ 2), compute

    S = ∑ A[i] · A[j]   for all 1 ≤ i < j ≤ n
    (sum of products of all unordered pairs)

Naive approach:
- Two nested loops
- Time complexity: O(n²)

Optimized O(n) approach:

Let
    total = ∑ A[k]   for k = 1 to n

Then
    S = (1/2) · ∑ A[i] · (total − A[i])   for i = 1 to n

Why this works:
- For a fixed index i, the value A[i] is multiplied with every other element:
      A[i]·A[1], A[i]·A[2], …, A[i]·A[n] (except A[i]·A[i])
- The sum of those other elements is (total − A[i])
- So contribution of index i is:
      A[i] · (total − A[i])

- Summing this over all i counts every unordered pair (i, j) twice:
      once as A[i]·A[j] and once as A[j]·A[i]

- Dividing by 2 removes this double counting

Final formula:
    S = (∑ A[i] · (total − A[i])) / 2

Time complexity: O(n)
Space complexity: O(1)

82. COORDINATE COMPRESSION

What coordinate compression does
--------------------------------
Maps large / negative / sparse values to small contiguous indices
while preserving relative order.

If x < y originally,
then compressed(x) < compressed(y).

Only ordering is preserved, not distances.


When coordinate compression is applicable
------------------------------------------
- Values are large (up to 1e18)
- Values are negative
- Values are sparse
- You only need ordering, not actual values
- Total number of DISTINCT values is manageable


Step 1: Collect all values in a set
-----------------------------------
Insert EVERY value that will ever be:
- accessed
- updated
- compared
- used as a boundary

Using a set guarantees:
- uniqueness
- sorted order

Example:
set<long long> st;

st.insert(x);


Step 2: Convert set → vector
----------------------------
Create a vector from the set to enable indexing.

vector<long long> coords;
for (auto x : st)
    coords.push_back(x);

Now:
- coords is sorted
- coords contains unique values
- coords[i] is the (i+1)-th smallest value


Step 3: Define 1-based indexing
-------------------------------
Mapping rule:

compressed(x) =
    lower_bound(coords.begin(), coords.end(), x)
    - coords.begin()
    + 1

So:
coords[0] → index 1
coords[1] → index 2
...
coords[k-1] → index k

Important Rules 
--------------------------------
- You MUST collect values BEFORE compression
- You MUST compress ALL values that will appear
- Do NOT compress values that appear only at runtime unless known beforehand
- Use lower_bound, not upper_bound
- Be consistent with 1-based indexing


Memory Constraint 
-----------------------------
Coordinate compression is feasible ONLY if:

number of UNIQUE values << available memory

Typical limits:
- ~1e6 unique values → always safe
- ~5e6 unique values → usually safe
- ~1e7 unique values → borderline
- >1e7 unique values → often NOT safe

Reason:
- Segment tree requires ~4 × N nodes
- Each node stores at least one 64-bit value
- Memory explodes quickly

Coordinate compression DOES NOT magically reduce memory usage.
It only makes indexing possible.

If the number of DISTINCT values is too large,
compression is useless and will TLE / MLE.

In such cases, use:
- Treap
- Ordered map
- Policy-based data structure
- Hashing (if ordering not needed)

83. Delting

What is Delting?
----------------
Delting is a technique used to avoid repeatedly updating all elements
of a data structure when a query adds the same value to every element.

Instead of modifying each element, we maintain a global offset.


When to Use Delting
-------------------
Use delting when:
- A query adds a value to ALL elements in a DS
- Other queries read or update individual elements
- Relative differences between elements matter more than absolute values


Core Idea
---------
Maintain an auxiliary variable:

aux = 0  (initially)

Invariant:
real_value = stored_value + aux

We NEVER store real values directly.


Operations
----------

1) Add x to all elements
-----------------------
Instead of updating all elements:

aux += x

Time: O(1)


2) Insert a value v
------------------
We store the adjusted value:

stored_value = v - aux

Insert stored_value into the DS.


3) Get / Query a value
----------------------
When a value is retrieved:

real_value = stored_value + aux


4) Update an existing element
-----------------------------
To add x to a specific element:

stored_value += x

(aux is NOT changed)


Why This Works
--------------
All elements share the same global offset.
Adding x to every element is equivalent to shifting the offset.

By storing values relative to aux:
- Global updates become O(1)
- Local updates remain correct
- Order comparisons are preserved


Important Properties
--------------------
- Comparisons between elements are preserved
- Differences between elements are unchanged
- Works with negative values
- Avoids O(n) mass updates

84. ========================================================
2D → 1D REDUCTION + TWO POINTERS (MONOTONIC RECTANGLES)
========================================================

CORE IDEA
---------
Any sub-rectangle is defined by:
    (left, right, top, bottom)

If we fix (left, right),
the problem reduces to choosing (top, bottom),
which is a 1D subarray problem on rows.

This allows us to replace a 2D search
with many 1D optimizations.


STEP 1: FIX ONE DIMENSION
------------------------
Choose one dimension to fix (usually columns).

For every column pair [L, R]:
    For each row i:
        row_cost[i] = contribution of row i in columns [L, R]

This "compresses" the rectangle into a 1D array.


STEP 2: DEFINE THE 1D PROPERTY
-----------------------------
Now we solve a problem on row_cost[]:

Examples:
- Find longest subarray with sum ≤ S
- Find max-length subarray satisfying a constraint
- Maximize (length × width)

The width is fixed (R - L + 1).
Only the height varies.


WHY THIS REDUCTION WORKS
-----------------------
Because the property of a rectangle over rows
can be expressed as a function over row_cost[].

Mathematically:
    rectangle_property(L, R, top, bottom)
        = f(row_cost[top..bottom])

This is the key condition for applying 2D → 1D.


========================================================
MONOTONICITY AND TWO POINTERS
========================================================

DEFINITION
----------
A property P over subarrays is:

- Subsegment monotonic if:
    If P holds for [l, r],
    then it holds for any [l', r'] ⊆ [l, r].

- Supersegment monotonic if:
    If P holds for [l, r],
    then it holds for any [l', r'] ⊇ [l, r].


WHY MONOTONICITY MATTERS
-----------------------
Monotonicity allows a sliding window:
- Expanding the window changes feasibility in ONE direction
- Shrinking reverses it

This guarantees:
- Two pointers move forward only
- Total operations are linear


COMMON CASE (MOST IMPORTANT)
----------------------------
Constraint of the form:
    cost ≤ limit

Where:
- cost increases when adding rows
- cost decreases when removing rows

This is SUBSEGMENT MONOTONIC.


========================================================
TWO POINTERS AFTER 2D → 1D
========================================================

Given fixed [L, R]:

Initialize:
    up = 1
    current_cost = 0

For down = 1 .. rows:
    add row_cost[down] to current_cost

    while current_cost > limit:
        remove row_cost[up] from current_cost
        up++

    // now [up, down] is the largest valid window ending at down
    update answer using (down - up + 1)


WHY THIS IS CORRECT
------------------
- Extending down increases cost → may violate constraint
- Moving up decreases cost → restores feasibility
- Once up moves forward, it never needs to move back

This relies entirely on monotonicity.


========================================================
HOW THIS APPLIES TO RECTANGLES
========================================================

Rectangle feasibility:
- Adding a row → adds cost
- Removing a row → removes cost

Thus:
- Feasibility is monotonic w.r.t vertical expansion
- Two pointers are valid


Final rectangle area:
    area = height × fixed_width


========================================================
WHEN THIS PATTERN APPLIES
========================================================

Use 2D → 1D + two pointers when:
- You need a sub-rectangle
- Constraint is additive over rows or columns
- Adding rows/columns worsens feasibility monotonically

Typical constraints:
- Sum ≤ K
- Count of bad cells ≤ K
- Cost ≤ budget
- At most K violations


========================================================
WHEN IT DOES NOT APPLY
========================================================

Two pointers FAIL if:
- Property is non-monotonic
- Feasibility can break and reappear by expanding
- Constraint depends on exact positions non-additively

Example:
- XOR constraints
- Exact equality requirements
- Non-contiguous interactions


========================================================
MENTAL CHECKLIST
========================================================

1) Can I fix two sides of the rectangle?
2) Can I compress the rest into 1D?
3) Does feasibility change monotonically when extending?
4) If yes → two pointers

If all answers are YES,
this pattern is almost certainly intended.

84. ========================================================
TOP-K ONLINE MAINTENANCE (TWO-SET TECHNIQUE)
========================================================

WHAT IT IS
----------
A technique to dynamically maintain the K best elements
(largest or smallest) from a changing set of elements,
supporting online insertions and deletions.

At every moment, the chosen K elements must be optimal.


WHEN IT IS USED
---------------
- You are allowed to select at most K elements
- Selection gives special benefit (free, discounted, upgraded)
- New elements arrive, old ones leave
- The optimal K must be maintained continuously


CORE IDEA
---------
Partition elements into two disjoint sets:

    S_best  → the current K best elements
    S_rest  → all remaining elements

Invariant:
- |S_best| ≤ K
- Every element in S_best is better than every element in S_rest


WHAT “BETTER” MEANS
-------------------
Depends on the problem:
- Larger value
- Smaller value
- Higher benefit
- Lower cost

Top-K = selecting elements with best priority.


INSERT OPERATION
----------------
When inserting element x:

1) If |S_best| < K:
       insert x into S_best
2) Else:
       let y = worst element in S_best
       if x is better than y:
           remove y from S_best
           insert y into S_rest
           insert x into S_best
       else:
           insert x into S_rest

This preserves the invariant.


DELETE OPERATION
----------------
When deleting element x:

1) If x ∈ S_best:
       remove x
       promote the best element from S_rest into S_best
2) Else:
       remove x from S_rest

Again, the invariant is restored.


WHY THIS IS CORRECT
-------------------
Greedy rule:
- The optimal solution always consists of the K best elements.

Exchange argument:
- If an element in S_rest is better than one in S_best,
  swapping them strictly improves the solution.

Therefore, maintaining this partition maintains optimality.


DATA STRUCTURE REQUIREMENTS
---------------------------
We must support:
- Insert
- Delete
- Find worst in S_best
- Find best in S_rest

Typical choices:
- Two multisets
- Two heaps with lazy deletion
- Balanced BSTs


TIME COMPLEXITY
---------------
Each operation:
    O(log N)

This is fast enough for online use.


COMMON APPLICATIONS
-------------------
- Selecting K largest / smallest values
- K coupons / free repairs
- K rows fully repaired
- K elements to ignore
- Maintaining K best candidates


MENTAL MODEL
------------
"There are K privileged slots.
They must always be occupied by the best possible elements."

85. ===============================
PITCHER POURING TECHNIQUE
===============================

1) WHAT THE TECHNIQUE IS (ABSTRACT VIEW)
---------------------------------------

Pitcher pouring is a STATE-MAINTENANCE technique.

You maintain a dynamically changing multiset while continuously preserving
some "balanced" invariant around a pivot or optimal value.

Instead of recomputing the answer from scratch after every update, you:
- Keep the data split into a SMALL NUMBER of containers ("pitchers")
- Move elements between them incrementally ("pouring")
- Ensure the invariant is restored after each local change

Key idea:
    The optimal solution moves slowly.
    So you can fix it with local repairs.

This is NOT a specific data structure.
It is a way of ORGANIZING updates.


2) CORE STRUCTURE
----------------

You maintain 2 (sometimes more) containers:

    LEFT  PITCHER  = elements contributing on one side of the optimum
    RIGHT PITCHER  = elements contributing on the other side

These containers can be:
- heaps
- multisets
- balanced BSTs
- indexed sets
- any structure that supports:
      insert
      delete
      extract boundary element

The containers are NOT symmetric:
Each one has a ROLE with respect to the invariant.


3) THE INVARIANT (MOST IMPORTANT PART)
-------------------------------------

You explicitly maintain a small set of rules that define correctness.

Typical invariants look like:
- All elements in LEFT <= all elements in RIGHT
- Size difference is bounded
- Cost contributions are separated cleanly
- One pitcher defines the "current optimal value"

The algorithm is nothing more than:
    After every update → restore invariants.

If you cannot clearly state the invariant,
you cannot use this technique.


4) POURING (REBALANCING)
-----------------------

"Pouring" means:
    Move boundary elements between pitchers
    until the invariant holds again.

Properties:
- Pouring is LOCAL
- Usually only O(1) elements move per update
- Pouring is deterministic once the invariant is defined

Important:
    Pouring does NOT depend on problem size.
    It depends on how much the optimum shifts.


5) WHY IT WORKS
---------------

Pitcher pouring relies on this property:

    Between consecutive states,
    the optimal solution changes slightly.

This is common when:
- You slide a window
- You move a pointer
- You add/remove one element at a time
- You expand/contract a range

Because the change is small,
the fix is also small.


6) WHEN TO USE IT
----------------

You should think of pitcher pouring when ALL of these hold:

1) You have a dynamic set (insert/delete over time)
2) You need to maintain an OPTIMAL value, not just existence
3) The optimal value depends on ORDER or PARTITIONING
4) Updates are incremental (1 element at a time)
5) Recomputing from scratch is too slow

Classic signals:
- Sliding window
- Two pointers
- Expanding rectangles / intervals
- Online queries where range endpoints move gradually


7) HOW TO IDENTIFY IT IN PROBLEMS
--------------------------------

Strong hints in problem statements:

- "Maintain the minimum cost as the window moves"
- "After each addition/removal, update the answer"
- "The optimal choice is the median / pivot / balance point"
- "Elements move in and out one by one"
- "The answer does not change drastically between steps"

If you can say:
    "If I knew the answer for the previous state,
     I could fix it quickly for the next state"

→ pitcher pouring applies.


8) RELATION TO SOLUTION BAGS
---------------------------

Solution bags:
- Partition solutions into a few buckets
- Maintain aggregate information per bucket

Pitcher pouring:
- A dynamic, ordered version of solution bags
- Buckets are ordered by contribution
- Elements can move between buckets

You can think of it as:
    Ordered solution bags + rebalancing


9) WHAT DATA STRUCTURES WORK
----------------------------

ANY structure that supports:
- insert
- delete (possibly arbitrary)
- extract min/max/boundary

Common choices:
- multiset / TreeSet
- heap + lazy deletion
- ordered statistic tree
- custom balanced structure

The technique is independent of the structure.
Choose based on removal requirements.


10) LIMITATIONS / WHEN IT FAILS
-------------------------------

Pitcher pouring does NOT work when:

- Optimal solution jumps wildly between updates
- Many elements need to move per update
- Invariant is global, not local
- Updates are batch-style, not incremental
- You cannot cleanly partition contributions

If pouring cost becomes O(n) per step,
the technique collapses.


11) MENTAL MODEL
----------------

Think of two physical pitchers of water.

Each update:
- You add/remove a small amount of water
- The water level tilts slightly
- You pour a little between pitchers
- Balance is restored

You NEVER empty and refill the pitchers.


12) SUMMARY
-----------

Pitcher pouring is:
- An invariant-driven technique
- For maintaining optimal values dynamically
- Using small local fixes instead of recomputation

Mastering it is about:
- Identifying slow-moving optima
- Designing the right invariant
- Choosing containers that make pouring cheap

If you understand the invariant,
the rest is mechanical.

85. A Greedy Proof Technique

For feasibility problems with repeated operations, first reinterpret each operation in terms of resources or units rather than raw numbers, and identify invariants such as parity, total sum, or monotonic measures. 

Use these invariants and extreme cases (like the largest element versus the rest) to derive necessary conditions for success, and also identify obvious dead-end states. 

Then assume these conditions hold and prove they are sufficient by defining a greedy move that is always legal under the conditions and reduces the problem. 

Show that the necessary conditions are preserved after each move and that any dead-end would contradict them, implying the process can never get stuck. Since progress is guaranteed and the process terminates, the target state must be reachable whenever the necessary conditions hold.

This technique is used for YES/NO problems where a process applies repeated operations and asks whether a target state is reachable.

i. When should you think of this technique?

Use this method if the problem has:

- Repeated operations on numbers/objects
- Each operation consumes or redistributes resources and affects multiple elements at once
- The order of operations does not matter
- The question is feasibility, not construction

ii. Step 1 -> Reinterpret the Question

Never reason directly on raw numbers.
Translate the operation into:
- units
- tokens
- resources
- supply v/s demand

Ask:
- what does on operation actually do?
- what is removed, transferred, or paired?


Example pattern:
    "Decrease two different elements by 1"
    → "Remove two units from different piles"

This abstraction reveals the structure of the problem.

iii. Step 2 -> Identify Invariants

An invariant is a quantity that:
- never changes, or
- changes in strictly fixed way per operation

Typical invariants:
- parity of sum (even / odd)
- sum modulo k
- monotonic measures (always decreases)

Invariants give NON-NEGOTIABLE constraints.

Example:
    Each operation removes exactly 2 units
    → sum parity is invariant

iv. Step 3 -> Derive Necessary Conditions

Ask: "In which situations is the goal clearly impossible?"

Techniques:

(A) Parity / modular constraints
    If each operation changes sum by k,
    final sum must be divisible by k.

(B) Extreme element analysis
    Compare the largest element to the rest.
    If one element demands more than all others can supply,
    the process must fail.

(C) Dead-end states
    Identify states where:
        - no operation is possible
        - goal is not yet achieved
    If such a state is reachable → impossible.

If ANY necessary condition fails → answer is NO.

v. Step 4 -> Hypothesize Sufficiency

Assume all necessary conditions hold.

Now the goal is to prove:
    "The process can always continue until completion."

Do NOT attempt full construction.
You only need to prove existence of valid moves.

vi. Step 5 -> Define a greedy move

Choose a move that:
- is legal whenever conditions hold
- simplifies the state

Common greedy choices:
- pick two largest non-zero elements
- pair a demand with any available supply
- reduce the most constrained elements first

The exact choice matters less than proving it always exists.

vii. Step 6 -> Prove Invariant Preservation

This is the core of sufficiency.

You must show:
    Necessary Conditions hold before the move
    ⇒
    Necessary Conditions hold after the move

Usually done by:
- bounding the maximum element
- comparing sum before / after
- showing inequalities are preserved

This prevents future failure.

viii. Step 7 -> Eliminate Dead-End States

Identify what a dead-end looks like.

Then prove:
    dead-end ⇒ violates a necessary condition

Therefore:
- dead-ends are unreachable
- the greedy process cannot get stuck

This replaces explicit simulation.

ix. Step 8 -> Conclude Sufficiency

Since:
- a valid move always exists
- necessary conditions are preserved
- the process strictly progresses

⇒ the target state must be reachable.

x. Final Decision

If any necessary condition fails → NO
Otherwise → YES

86. MAXIMUM MARGINAL GAIN (DISCRETE)

Problem Type
-------------
We are minimizing a sum of independent convex terms:
    S = Σ f_i(x_i)
subject to a fixed number of unit operations (±1) on variables.

Each operation locally changes exactly one term f_i.

Key Insight
------------
At any step, the best operation is the one that gives the
maximum immediate decrease in the total objective.

For a single variable x:
    Δ = f(x ± 1) − f(x)
Choose the move with the most negative Δ (largest drop).

Interpretation:
This is a discrete version of “follow the largest derivative”.

Application to (A[i] − B[i])²
------------------------------
Let d_i = A[i] − B[i].

Error contribution:
    f_i = d_i²

Effect of changing A[i] or B[i] by ±1:
    d_i → d_i ± 1

Change in error:
    (d_i ± 1)² − d_i² = ±2d_i + 1

Magnitude of improvement depends on |d_i|.
Hence:
    The larger |d_i| is, the more beneficial it is to reduce it.

Greedy Rule
------------
Repeatedly:
1. Pick index i with maximum |d_i|
2. Apply one operation to reduce |d_i| by 1
3. Update d_i and repeat until operations are exhausted

Implementation Pattern
----------------------
- Use a max-heap keyed by |d_i|
- Each operation:
    pop max
    update d_i toward zero
    push updated value back

Why Greedy Is Correct
---------------------
1. Each term is independent and convex
2. Marginal gain strictly decreases as |d_i| decreases
3. No operation affects any other term
4. Exchange argument:
   Any optimal solution can be rearranged to apply reductions
   to larger |d_i| first without increasing total cost

When to Use This Strategy
------------------------
- Objective is a sum of convex per-element costs
- Operations are discrete and local (±1)
- Fixed number of operations
- Want to minimize total cost

Mental Model
------------
"Always spend your limited operations where they buy you the
largest immediate reduction in error"

87. Counting number of unique elements with a Segment Tree

If the values in the array lie in a small bounded range such as 0 ≤ v ≤ 63, we can efficiently maintain the number of distinct elements in any segment using a segment tree with bitmasks. Instead of storing frequencies, each node stores a 64-bit integer mask representing which values appear in its segment. The i-th bit of the mask is set if the value i occurs at least once in that segment.

For a leaf node corresponding to element a[i], the mask is simply (1 << a[i]). For an internal node, the mask is computed as the bitwise OR of the masks of its two children, because a value appears in the parent segment if it appears in either child.

To answer a query for a range [l, r], we combine the masks of the segment tree nodes that cover the range using bitwise OR. The resulting mask represents all values present in that range. The number of unique elements is then obtained by counting the number of set bits (popcount) in the final mask.

Point updates are handled by updating the mask at the leaf and recomputing masks along the path to the root using OR. Since each node stores only a fixed-size 64-bit mask, both queries and updates run in O(log n) time with very small constant factors.

Range updates are a bit trickier to handle and I've done a question regarding this in random/newYearTree.cpp

88. Technique: Logarithm Trick for Products

Idea

Multiplication grows extremely fast and easily overflows.  
Use logarithms to convert multiplication into addition.

Key identities:

    log(a * b) = log(a) + log(b)
    log(a / b) = log(a) - log(b)

Thus we store:

    log(value)

instead of:

    value

------------------------------------------------------------

Product Queries

If

    product = a1 * a2 * ... * ak

then

    log(product) = log(a1) + log(a2) + ... + log(ak)

So a product query becomes a **sum query over logs**.

Example:

    store log(arr[i])

    product(l, r)
        = exp(sum(log(arr[l..r])))

Often the exp step is avoided if only comparison is needed.

------------------------------------------------------------

Ratio Queries

For

    ratio = A / B

take logs:

    log(ratio) = log(A) - log(B)

Thus:

    ratio = exp(log(A) - log(B))

If we only need to check bounds:

    if log(A) - log(B) > log(LIMIT)
        ratio > LIMIT

------------------------------------------------------------

Comparing Products

To compare

    a*b*c  vs  x*y*z

compare

    log(a)+log(b)+log(c)
    log(x)+log(y)+log(z)

No multiplication needed.

------------------------------------------------------------

Typical Usage

Use this technique when problems involve:

    • large products
    • ratios of products
    • product range queries
    • comparing products
    • multiplication of many values

Implementation pattern:

    store: log(value)
    product → sum(logs)
    ratio   → difference of sums
    recover value → exp(result) (only if safe)

Constraints

    value must be > 0  (log undefined for 0)

89. Techniques for counting the number of occurrences of x in a subarray

i. Problem 1

Given an array A[1, n], answer queries:

    count how many times value x appears in subarray [l, r]

where x can be different for each query.

Solution:

1. Create a hash map f such that f[x] stores all the indices of occurences of x in A[1, n].

unordered_map<int, vector<int>> f;
for (int i = 1; i <= n; i++) f[A[i]].push_back(i);

2. To count the occurrences of x in [l, r]:

- Look at the vector f[x]
- Find first position >= l
- Find first position > r
- Their difference is the answer

Thus, answer = upper_bound(f[x].begin(), f[x].end(), r) - lower_bound(f[x].begin(), f[x].end(), l);

Complexity:

1. Preprocessing: O(n)
2. Query O(log(n)) (worst case)

ii. Problem 2

Maintain an array A[1, n] supporting:

    - Update: change A[i] to a new value
    - Query: count occurrences of x in subarray A[l, r]

where value x may change per query.

Solution:

1. Instead of storing positions in vectors, store them in ordered sets so that insertions and deletions can be handled efficiently.

Create a hash map f such that f[x] is a set that stores all indices where value x occurs.

unordered_map<int, set<int>> f;

for (int i = 1; i <= n; i++)
    f[A[i]].insert(i);


2. To count the occurrences of x in [l, r]:

- Look at the set f[x]
- Find the first element >= l
- Find the first element > r
- The number of elements between them is the answer

Thus,

auto L = f[x].lower_bound(l);
auto R = f[x].upper_bound(r);

answer = distance(L, R);


3. To perform an update A[i] → new_value:

Let old_value = A[i]

Step 1: Remove index i from the set of the old value

    f[old_value].erase(i);

Step 2: Insert index i into the set of the new value

    f[new_value].insert(i);

Step 3: Update the array

    A[i] = new_value;


Complexity:

1. Preprocessing: O(n log n)
2. Update: O(log n)
3. Query: O(log n + k)

where k is the number of occurrences returned.
Worst case query complexity is O(n), but in practice it is typically small.

iii. Problem 3

Maintain an array A[1, n] supporting:

    - Update: change A[i] to a new value
    - Query: count occurrences of x in subarray A[l, r]

where value x may change per query.

Solution (Segment Tree / Merge Sort Tree):

1. Build a segment tree where each node stores a sorted list of the values present in its segment.

Thus each node maintains:
    tree[node] = sorted list of A[l..r]

Build rule:

- If node represents a single element:
    tree[node] = { A[i] }

- Otherwise merge the sorted vectors of its children:
    tree[node] = merge(tree[left_child], tree[right_child])


2. To count occurrences of x in [l, r]:

Traverse the segment tree nodes that cover the query range.

For each visited node:

- Its vector contains all values in that segment.
- Count occurrences of x using binary search.

Thus,

    count =
        upper_bound(tree[node].begin(), tree[node].end(), x)
      - lower_bound(tree[node].begin(), tree[node].end(), x)

Sum the results over all visited nodes.


3. To perform an update A[i] → new_value:

Update the leaf corresponding to position i.

Then while moving up the tree:

- Remove the old value from the node's vector
- Insert the new value while keeping the vector sorted


Complexity:

1. Preprocessing:  O(n log n)
2. Query:          O(log² n)

        (log n nodes × log n binary search)

3. Update:         O(log² n)
4. Memory: O(n log n)


Key Idea:

Store sorted values in each segment tree node so that frequency queries
inside any segment can be answered using binary search.

This method generalizes well to many problems where we need to query
value-based properties inside a range.
